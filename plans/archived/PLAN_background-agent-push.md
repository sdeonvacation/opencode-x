# Plan: Background Agent Push (Leader+D Detach)

## Context

Today, while opencode's TUI is mid-run, the user's prompt is blocked until the assistant finishes. Long agent runs idle the operator. claude-code solves this with a "push to background" chord: detach the running task, free the prompt, notify on completion.

opencode already has every primitive needed (`BackgroundJob.Service`, `SessionRunState/Runner`, `TuiEvent.BackgroundTaskUpdate`, `ToastShow`, leader-key chord pattern in `tui/context/keybind.tsx`, `prompt-reminders.ts`). What's missing is a glue RPC + chord that turns an in-flight foreground run into a tracked `BackgroundJob` *without interrupting the underlying fiber*.

Outcome: pressing `<leader> d` during a busy session unblocks the TUI input, leaves the agent running in-process, and surfaces a toast + next-turn `<system-reminder>` when the detached run finishes. Primary agent sees the completion via the reminder on its next turn — no autonomous resumption.

## Approach: Detach-by-Wrap

Reuse the existing `Runner` per session (`packages/opencode/src/session/run-state.ts:35-92`). The runner already owns the fiber and a `Deferred`-equivalent state machine. We don't move or rewrap the fiber. Instead we register a new `BackgroundJob` whose `run` Effect awaits the existing runner state, and we tell the TUI to treat that session as idle.

### 1. Server: `session.background` RPC

- Add `SessionPrompt.background(sessionID)` next to `SessionPrompt.cancel` (`packages/opencode/src/session/prompt.ts:120-122, 1979`).
  - Behaviour: look up the runner; if `state._tag !== "Running"` return false. Else register a `BackgroundJob.start({id: sessionID, type: "session.background", title: <session title>, metadata: {sessionID}, run})` where `run` is an Effect that awaits the runner's pending `Deferred` / `Fiber.await(handle.fiber)` and resolves to a one-line summary (assistant message text, last 200 chars). On success/failure, `BackgroundJob.start`'s built-in `matchCauseEffect` already maps to `completed`/`error` (`background/job.ts:start` body).
  - Add `Effect.tap` step that publishes `TuiEvent.BackgroundTaskUpdate {sessionID, taskID:sessionID, title, state}` and `TuiEvent.ToastShow` — same pattern the `task` tool uses today (`tool/task.ts:inject` block).
  - Add `Effect.tap` step that calls `SessionPromptReminders.queueDetachedCompletion(sessionID, summary, state)` so the next user turn sees a `<system-reminder>` with the result.
- Expose `SessionRunState.snapshot(sessionID)` that returns `{busy, awaitDone}` (or expose `runner(sessionID)` non-mutating getter). Needed so `SessionPrompt.background` can subscribe to fiber completion without taking ownership. Reuse the existing `runners` Map; add a read-only `peek(sessionID)` interface.

- Add `POST /:sessionID/background` route in `packages/opencode/src/server/routes/session.ts` (mirror of the abort route at line 415, operationId `session.background`). Calls `SessionPrompt.background`.

### 2. SDK regeneration

- SDK is auto-generated by hey-api. After adding the route, regenerate: `bun run` whatever script lives under `packages/sdk/js` (verify via `package.json` scripts in `packages/sdk/js/package.json`). New method appears as `client.session.background()` on `packages/sdk/js/src/v2/gen/sdk.gen.ts` (mirror of the existing `abort` at line 2114).

### 3. TUI: chord + state

- `packages/opencode/src/cli/cmd/tui/component/prompt/index.tsx`:
  - Add a new command entry next to `session.interrupt` (lines ~290-378): `value: "session.background"`, `keybind: "session_background"`, `enabled: status().type !== "idle" && !detachedSet.has(sessionID)`.
  - `onSelect` handler: call `sdk.client.session.background({sessionID})`, then add `sessionID` to a TUI-local detached-set signal, clear dialog. **No counter, no decay timer** — leader-chord already provides the gating + 2s timeout (`tui/context/keybind.tsx:14-30`).
- The detached-set signal is a `createSignal<Set<SessionID>>` at module scope in this component (or hoisted into `context/sync.tsx` if any other component needs it). When a session is in the set, the prompt input treats it as `idle` (input unblocked). When `session.status` event reports `idle` for that session (server finished), remove from set. When user submits a new prompt for that session, remove from set so subsequent detach works again.
- Existing `event.on(TuiEvent.BackgroundTaskUpdate.type, ...)` block at `prompt/index.tsx:209` already increments `bgTasks`. Reuse — `state: "running"` events fire when we publish from the server. On `completed`/`error`, the count decrements and a toast is shown by the existing event listener pipeline (`effect/app-event-listeners.ts:99` already wires `ToastShow`).

### 4. TUI: keybind

- `packages/opencode/src/cli/cmd/tui/context/keybind.tsx` — keybinds come from `TuiConfig.Info.keybinds`. Default `session_background` → `<leader>+d`. Add wherever defaults are seeded (search for `session_interrupt` default, add a sibling). Update `TuiConfig` schema to include the new key.
- The existing leader-chord wiring (`leader(active)` function in `keybind.tsx`) handles the 2s decay automatically — no new chord state machine needed.

### 5. Reminder injection

Detach is **same-session**: the `sessionID` of the detached run equals the user's foreground session. The detached run's full transcript is already in the session's message history. The reminder's job is just to flag completion to the next assistant turn — not to ship the full output.

- New file `packages/opencode/src/session/detached-notes.ts`:
  - Module-level `Map<SessionID, Array<{state: "completed"|"error"|"cancelled", summary: string}>>`.
  - Exports `queue(sessionID, state, summary)` and `drain(sessionID): Note[]`.
- `packages/opencode/src/session/prompt-reminders.ts`:
  - Import `DetachedNotes` from the new file.
  - Inside `insertReminders`, just before the final `return { messages, changed }`, drain pending notes for the session. For each, append a synthetic text part to the last user message:
    ```
    <system-reminder>
    Background task completed. The prior assistant turn (which the user detached) finished with status: <state>. Its full output is the most recent assistant message in this conversation. Continue based on user's next instruction. Do not autonomously act on the result unless the user asks.
    </system-reminder>
    ```
  - Drain on read so each note fires exactly once. If user never returns to the session, notes accumulate; on next submission, all drain at once (bounded by detach count for that session).
- Notification policy: **passive**. Primary agent does not auto-resume. No synthetic prompt is fired. The reminder is informational only; the user's next prompt drives the next turn. Auto-resume is a future config-gated extension, out of scope for v1.

### 6. Failure / edge cases

| Case | Handling |
| --- | --- |
| Detach when session is `idle` | RPC returns `false`; TUI no-ops (chord disabled by `enabled` gate too). |
| User hits `Ctrl+C` after detach | Existing `session.interrupt` chord still calls `session.abort` → `SessionPrompt.cancel` → `SessionRunState.cancel` interrupts the underlying runner; the `BackgroundJob` resolves as `cancelled` via `matchCauseEffect`'s `Cause.hasInterruptsOnly` branch (`background/job.ts:start`). |
| Quit while detached | `BackgroundJob` is forked into the layer scope; on app shutdown the scope finalizer interrupts. Already handled. Final assistant message persistence pattern from `subtask-handler.ts:Effect.onInterrupt` (lines around `assistantMessage.finish = "tool-calls"`) applies in `processor.ts` already; no additional code. |
| Double detach | Idempotent: `BackgroundJob.start` short-circuits if `existing?.info.status === "running"` and id collides. |
| Two different sessions detached | Independent `BackgroundJob` entries keyed by `sessionID`. Existing `model_concurrency` guards still apply on the underlying runs. |

### 7. Rollout gate

Reuse `Flag.OPENCODE_EXPERIMENTAL_BACKGROUND_SUBAGENTS` (`packages/opencode/src/flag/flag.ts:78`). The new RPC throws if the flag is off; the TUI chord is registered always but the RPC short-circuits, so users on stable get a toast "background detach not enabled".

## UX behavior

**Trigger**
- Session busy. User presses `<leader>` (existing TUI leader key — set by `keybinds.leader` config). Within 2s: presses `d`. RPC fires. Single chord, no counter, no second confirmation.
- Leader timeout (2s) handled by existing `keybind.tsx:leader()` function — if user doesn't press `d` in time, leader resets and original focus is restored.

**Idle / already-detached**
- `enabled: status().type !== "idle" && !detachedSet.has(sessionID)`. Chord is disabled when session is idle or already detached. No-op press.

**Immediately after detach**
- Prompt input unblocks. Cursor active.
- Status indicator switches to `↓ background` marker (distinct from idle).
- `bgTasks` badge increments (existing wiring at `prompt/index.tsx:209`).
- Toast: `"Detached. Will notify on completion."`

**Concurrent input on same session**
- v1: TUI blocks new submissions to a detached session — input is enabled but submission is short-circuited with a toast `"Session has detached run; wait for completion or switch sessions."` Reasoning: server-side `Runner.ensureRunning` would queue a second prompt behind the running one, leading to surprising serialization. Cleaner UX to force the user to switch sessions.
- User can switch to other sessions freely; detached one keeps running.

**Completion**
- Detached run finishes (success / error / cancelled).
- `BackgroundJob.finish` resolves; server publishes `BackgroundTaskUpdate` + `ToastShow`.
- TUI toast: `"Background task complete: <session title>"` (or "failed" / "cancelled").
- `bgTasks` count decrements. Detached marker clears for that session.
- `DetachedNotes.queue(sessionID, state, summary)` writes a pending note.

**Next user submission to that session**
- `insertReminders` drains the note and appends one `<system-reminder>` text part to the user message before model call. Agent reads it as context alongside its scrollback (which already contains the full detached output as the most recent assistant message).
- Agent does **not** auto-resume. It only acts if the user's prompt asks something related.

**Quit while detached**
- App shutdown interrupts the layer scope → runner fiber interrupted → `Effect.onInterrupt` in `processor.ts` flushes final assistant message → `BackgroundJob` resolves as `cancelled`. No silent data loss.

**Manual interrupt after detach**
- `session.interrupt` chord still calls `session.abort` → `SessionPrompt.cancel` → `SessionRunState.cancel` interrupts the runner → `BackgroundJob` resolves `cancelled` via `Cause.hasInterruptsOnly` branch in `background/job.ts:start` → toast `"Background task cancelled"`.

**Failure mode UX**
- Detach RPC fails (flag off, no runner found): toast `"Cannot detach: <reason>"`. Detached-set unchanged. No silent failure.

## Primary agent notification

Two channels, different audiences:

1. **TUI toast (human-facing)** — `BackgroundTaskUpdate` + `ToastShow` events, identical path used by the `task` tool's bg branch (`tool/task.ts:inject`). Visible to operator. Not seen by LLM.
2. **`<system-reminder>` injection (agent-facing)** — drained from `detached-notes.ts` Map by `insertReminders` on the next user prompt. Becomes a text part on the user message before the model call.

**Does primary agent see the detached output?**
Yes — full output, via normal scrollback. Detach is same-session, so the detached run's final assistant message is already in the session's message history when primary's next turn fires. The reminder is just a flag pointing at it.

**Does primary act proactively?**
No. v1 is **passive notification only** — no synthetic auto-prompt, no agent resumption, no tool auto-call. Reminder explicitly says "Do not autonomously act on the result unless the user asks." Reasoning:
- User detached to free attention; auto-resume contradicts intent.
- claude-code parity: completion notifies, parent acts on next user turn.
- Easy to layer auto-resume later behind a config key without breaking v1 contract.

**`task_status` tool fallback**
For the future cross-session detach case (e.g. user detaches a subagent in a separate session), agents can already call `task_status({task_id})` to fetch full `BackgroundJob.Info` including `output`/`error` text. No new tool needed; existing tool covers the introspection path.

## Provider caching safety

Anthropic prompt cache hits require byte-identical prefix of `system` + tool defs + message history. One divergent byte → all subsequent tokens uncached. This feature must not perturb that prefix.

**Risk audit**

| Vector | Risk | Mitigation |
| --- | --- | --- |
| `<system-reminder>` injection | Could divergence cache if added to system prompt or older messages | Append only to the *new* user message (post-prefix boundary). Mirror existing plan-mode reminder placement in `prompt-reminders.ts`. |
| Re-injection on retries | If reminder reappears in persisted history on subsequent turns, message history forks → all later turns uncached | Drain-on-read. Reminder appears in exactly one persisted user message ever. Test it. |
| Detached state in system prompt | Adding "session is detached" / `bgTasks` count into system prompt = mid-conversation prefix change | Detached state is **TUI-local only**. Server-side message construction is identical whether or not user pressed leader+d. No `<background-running>` markers in system prompt or transcript. |
| `processor.ts` / `prompt.ts` perf path | Edits to message construction or cache-control markers would degrade caching | Zero edits. `SessionPrompt.background(...)` is a new sibling export. Does not touch `prompt`/`complete`/`loop`/`shell`/`command` bodies or `processor.ts` (lines 185+ `aborted` semantics). |
| `finish` field on cancelled detached run | Cancelled vs natural completion writes different `finish` value on the assistant message; if serialized to model = prefix change | Verify during impl: `MessageV2.WithParts` model serialization uses `parts` array, not `finish` metadata. Audit the model-facing serializer; assert `finish` is metadata-only. |
| `Effect.onInterrupt` flush on quit | Existing path — same bytes whether detached or not | No-op. |
| Cross-session detach interaction | Independent sessions = independent prefixes | No interaction. |
| Bus events / toasts | Pure TUI side effects | Not in model-facing prompt. |

**Guardrails (must hold)**

- Reminder text appended to user message only. Never to system prompt, never to assistant messages, never to earlier user messages.
- Drain on first read. Assert via test that re-invoking `insertReminders` on the same persisted user message does not re-append.
- Reminder content is **deterministic** given `(state, sessionID)`. Fixed template. No timestamps, counters, durations, or other volatile fields. Same `(state, sessionID)` always produces byte-identical reminder text.
- No new fields added to `MessageV2` schema or any other type that gets serialized into the model transcript.
- TUI-side `detachedSet` signal must not flow into any server-side request body.

**Cache regression test**

Add a test that:
1. Runs a session through one user turn (control). Captures the exact bytes of the assistant API request body for turn 2.
2. In a second run, detaches mid-turn-1, lets it complete, drains reminder, then runs turn 2.
3. Asserts: bytes of system prompt + tool defs + messages[0..N-1] (everything before turn 2's user message) are **byte-identical** between control and detached runs.
4. The new user message in run 2 differs (contains the reminder text part). Diff must be confined to that single message.

Place the test in `packages/opencode/test/session/background-detach-cache.test.ts`. Use existing test fixtures for prompt construction.

## Upstream-rebase safety

This repo (`opencode-x`) is a fork of upstream `opencode`. Future rebases will pull conflicting changes in the same hot files (`session/prompt.ts`, `routes/session.ts`, `tui/component/prompt/index.tsx`). Minimize blast radius:

- **No edits to existing function bodies in upstream files when avoidable.** Add new sibling functions/exports rather than modifying existing ones:
  - `SessionPrompt.background(...)` is a *new* exported function. Do not alter `SessionPrompt.cancel`.
  - `POST /:sessionID/background` is a *new* route block. Do not modify the existing abort route at line 415.
  - `queueDetachedCompletion` is a *new* export from `prompt-reminders.ts`. Inside `insertReminders`, the only modification is appending a single new block guarded by `if (pending.length)` — placed at the end of the function so upstream additions land before it. Anchor the insertion at the `return { messages, changed }` line (last line) — most stable diff anchor.
  - In `prompt/index.tsx`, the new chord command is a *new* entry in the commands array, placed immediately after `session.interrupt`. No edits to existing entries.
- **Isolate fork-specific glue in new files where possible.** Put `queueDetachedCompletion`'s pending-notes Map and helpers in a new file `packages/opencode/src/session/detached-notes.ts`; `prompt-reminders.ts` only imports + drains it. One-line import + one-line drain in upstream file = trivial conflict resolution.
- **Don't touch `run-state.ts` interface ordering.** `peek(sessionID)` is added as the *last* field in `Interface` and the *last* element in `Service.of({...})`. Upstream additions typically append; trailing-position adds rebase cleanly.
- **TuiConfig schema change is unavoidable** (new keybind key). Add `session_background` at the *end* of the keybind keys block, alphabetical order be damned — trailing position survives upstream additions cleanly.
- **SDK regen output is fully derived.** Don't hand-edit `packages/sdk/js/src/v2/gen/*`. Always regen from the route. Conflicts in generated files are resolved by re-running codegen after the rebase, never by manual merge.
- **Mark fork-only sections.** Wrap the new RPC body, the new route block, and the chord-command block with a comment marker like `// fork: background-detach (#FORK)` so a rebase author can grep for the feature surface and re-apply if upstream restructures the area.
- **Atomic commit boundary.** Land this work as a single commit (or a tightly-scoped commit per file) so a future `git revert` or `git cherry-pick` against upstream's main is mechanical.
- **Avoid renaming or restructuring upstream symbols.** Even if `cancel`/`abort` naming is inconsistent, leave it. New names use the `background` prefix consistently.
- **Test files separate.** Put new tests in a new file (`test/session/background-detach.test.ts`) rather than appending to existing test files; upstream test additions in same file produce nuisance conflicts.

Pre-rebase checklist for the operator:
1. `git log --oneline upstream/main..HEAD -- packages/opencode/src/session/prompt.ts` — identify all fork commits in hot files.
2. After rebase, regenerate SDK: `bun run --cwd packages/sdk/js generate`.
3. Re-run the verification suite below; the chord + RPC surface is the canary for whether the rebase silently broke detach.

## Out of scope (v1)

- `Ctrl+X Ctrl+K` stop-all-bg chord.
- Result-file path propagation for post-compaction recovery.
- `/bg` slash alias.
- Server-side `detached` status variant (chose TUI-local store per requirement clarification).
